# 02 - Architecture (Inside-Out Audit)

## What Code Shows

### Entry Point Flow
```
main.tsx
  -> GoogleOAuthProvider (wraps entire app)
    -> App.tsx
      -> QueryClientProvider (React Query)
        -> TooltipProvider (Radix)
          -> Toaster + Sonner (dual toast systems)
            -> AuthProvider (context)
              -> WorkspaceProvider (context, blocking gate)
                -> BrowserRouter
                  -> SidebarStateProvider (context)
                    -> AppErrorBoundary (class component)
                      -> Suspense
                        -> Routes
```

### Provider Hierarchy (outermost to innermost)
1. `GoogleOAuthProvider` - Google OAuth SDK
2. `QueryClientProvider` - React Query cache
3. `TooltipProvider` - Radix tooltip context
4. `AuthProvider` - JWT auth, user session
5. `WorkspaceProvider` - Active workspace, blocking gate
6. `BrowserRouter` - Client routing
7. `SidebarStateProvider` - Sidebar collapse state
8. `NovuProviderWrapper` - Notification SDK (only wraps AppLayout routes)

### Data Flow Pattern
1. **API Layer**: `ApiClient` (fetch wrapper) -> `services.*` (typed methods) -> `hooks.*` (React Query)
2. **Real-time**: `socketService` (Socket.IO singleton) -> `use-*-socket.ts` hooks -> component state
3. **Auth Flow**: JWT in localStorage -> `ApiClient.setToken()` -> Authorization header on all requests
4. **Workspace Scoping**: `workspace-id` header injected via interceptor on every request

### Two API Backends
- **apiClient**: Points to `VITE_DATA_LIST_SERVICE_BASE_URL` (list-service.icustomer.ai/api/v1) - Core data operations
- **gatewayClient**: Points to `VITE_GATEWAY_SERVICE_BASE_URL` (prod-aloop-api-gateway.icustomer.ai) - Integrations, OAuth, audience sync

### Feature Module Pattern
Each feature folder follows a consistent structure:
```
features/{name}/
  index.ts          - Barrel exports
  types.ts          - Domain types
  services.ts       - API calls
  stubs.ts          - Mock data (enrichment, batch-enrichment, grid)
  components/       - Feature-specific components
  hooks/            - Feature-specific hooks
  utils/            - Feature-specific utilities
```

## Patterns
- **Lazy loading**: All 34 page components use `React.lazy()` with a shared Suspense boundary
- **Service objects**: Stateless service objects (not classes) that call apiClient/gatewayClient
- **React Query everywhere**: All server state managed through React Query; no Redux or Zustand
- **Socket singleton**: Single SocketService instance shared across the app; hooks register/deregister listeners
- **Interceptor pattern**: Both API clients use request interceptors for workspace-id and user-id headers
- **Dual-FE awareness**: Multiple service methods include `x-client-origin` header and origin-based return URLs for Stripe, indicating a migration period where old and new frontends coexist

## Undocumented
- The `enrichment` and `batch-enrichment` features have nearly identical file structures (types.ts, services.ts, stubs.ts, hooks/*) suggesting a refactor that duplicated rather than consolidated
- The `data/` directory exists at `src/data/` but its contents were not fully explored
- `stubs.ts` files exist in enrichment, batch-enrichment, and grid features, suggesting mock data is available for development
- The `Vite proxy` configuration in `vite.config.ts` reveals the real backend topology: list-service for `/api/v1`, gateway for `/gateway`, and WebSocket for `/socket.io`

## Concerns
- The `src/api/services.ts` file is 1400+ lines containing 14 separate service objects. This is a maintenance concern; feature-level services (like `live-audiences/services.ts`) show the better pattern
- `audiencesService` in services.ts is fully deprecated but not removed
- No barrel exports for pages; App.tsx imports each page individually
- The `AppErrorBoundary` is a class component (required by React), but it sits inside the function component tree awkwardly between SidebarStateProvider and Suspense
- QueryClient is configured with `retry: 1` and `refetchOnWindowFocus: false` globally, which means stale data can persist longer than expected
