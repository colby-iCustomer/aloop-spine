# 05 - API Contracts (Inside-Out Audit)

## What Code Shows

The app communicates with two backend services via two separate HTTP clients.

### API Clients

**apiClient** - Data list service (`VITE_DATA_LIST_SERVICE_BASE_URL`)
- Dev proxy: `/api/v1` -> `https://list-service.icustomer.ai`
- Handles: auth, datasets, lists, enrichment, validation, normalization, payments, referrals, settings, webhooks, onboarding

**gatewayClient** - API gateway (`VITE_GATEWAY_SERVICE_BASE_URL`)
- Dev proxy: `/gateway` -> `https://prod-aloop-api-gateway.icustomer.ai` (path rewrite strips `/gateway`)
- Handles: integrations, OAuth, audience sync, export flows

## Patterns
- Consistent response unwrapping: `(res as any)?.data ?? res` pattern used everywhere
- Gateway responses need double unwrap: `response?.data?.data ?? response?.data`
- All mutations invalidate related React Query cache keys
- `skipAuth: true` flag for public endpoints (login, register, validate, etc.)
- Dual-FE support pattern: `x-client-origin` header and `window.location.origin` in redirect URLs

## Undocumented
- The socket has a listener registry that survives disconnect/reconnect cycles (for token refresh)
- Template view uses untyped socket events not in SocketEventMap
- `_systemProviderIdCache` in prospectingService caches the Apollo system provider ID for the session
- Connection providers (Salesforce, Instantly, Marketo) use a separate `/connections/*` endpoint pattern

## Concerns
- The `services.ts` file contains 14+ service objects in a single 1400+ line file
- Gateway response unwrapping is defensive but inconsistent; some use `unwrapGatewayResponse<T>()`, others do inline unwrapping
- Socket event types are split between typed (SocketEventMap) and untyped (onAny) with no central registry for the untyped ones
- Several deprecated service methods and hooks remain in the codebase with @deprecated tags but no removal timeline
