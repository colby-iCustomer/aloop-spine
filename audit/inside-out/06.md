# 06 - State Management (Inside-Out Audit)

## What Code Shows

### State Architecture Overview
The app uses a three-layer state model:
1. **React Context** - Auth and workspace state (user session, active workspace)
2. **React Query** - All server/API state (lists, folders, integrations, credits, etc.)
3. **Socket.IO** - Real-time event-driven state updates (enrichment progress, validation, etc.)

No Redux, Zustand, Jotai, or other external state libraries are used.

### React Contexts

**AuthContext** (`src/contexts/AuthContext.tsx`)
- State: `{ user: UserProfile | null, isAuthenticated: boolean, isLoading: boolean }`
- Actions: `login`, `signup`, `loginWithGoogle`, `loginWithMicrosoft`, `logout`, `updateProfile`, `setSession`
- Persistence: JWT stored in `localStorage.jwt_token`, user ID in `localStorage.user_id`
- Token refresh: Handled by ApiClient (401 -> POST /auth/refresh -> retry)
- Socket lifecycle: Calls `useSocketConnection(token, userId)` to connect/disconnect socket on auth changes
- Dev bypass: `dev-bypass-token` in localStorage skips auth and uses a mock user
- HMR safe: Uses global context key `__AUTH_CONTEXT__` to persist across hot reloads

**WorkspaceContext** (`src/contexts/WorkspaceContext.tsx`)
- State: `{ workspaces: Workspace[], currentWorkspace: Workspace | null, defaultWorkspaceId: string | null, isWorkspaceReady: boolean, isWorkspaceSwitching: boolean, workspaceError: string | null }`
- Actions: `switchWorkspace`, `setDefaultWorkspace`, `retryWorkspaceLoad`
- Persistence: `localStorage.lastUsedWorkspaceId`, `localStorage.defaultWorkspaceId`
- Blocking gate: Renders loading UI until workspace is resolved (prevents routes from rendering with no workspace)
- Resolution priority: lastUsedWorkspaceId -> defaultWorkspaceId -> API isDefault -> first workspace
- Legacy migration: Converts old `currentWorkspaceId` key to `lastUsedWorkspaceId`
- On switch: Clears all React Query cache, redirects resource pages to home

**SidebarStateProvider** (`src/components/layout/SidebarContext.tsx`)
- State: `{ collapsed: boolean }`
- Actions: `toggleSidebar`
- Simple toggle; no persistence (resets on refresh)

**TemplateHeaderContext** (`src/features/templates/contexts/TemplateHeaderContext.tsx`)
- Feature-level context for template view header state
- Scoped to the TemplateViewPage

  retry: 1

**Query Keys** (`src/api/hooks.ts:75-112`)
Centralized key definitions for cache management:
- profile, integrations, audiences, campaigns, folders, lists, listData, workspace, team, apiKeys, creditBreakdown, list, recentSheets, columnConfigs, subscription, referralStats, referralCode, userReferrals, myReferrer, couponStats, coupons, pixelStatus, tutorials, tutorialDocuments, templateMappings, allTemplateMappings, templateRegistry, onboardingProgress, salesforceAccounts, salesforceFields

**Stale Times:**
- Profile: 5 min
- Integrations: 2 min
- Folders: 2 min
- Audiences: 30 sec
- Campaigns: 60 sec
- OTP status: 10 sec polling
- Most others: default (0, always stale)

**Cache Invalidation Patterns:**
- Mutations invalidate related query keys via `qc.invalidateQueries()`
- Workspace switch: `queryClient.clear()` (nuclear - clears everything)
- User initialization: `queryClient.refetchQueries()` on folders key

### Socket-Driven State Updates

**Hooks that manage socket state:**
- `use-socket-connection.ts` - Lifecycle (connect on auth, disconnect on logout)
- `use-dataset-socket.ts` - Subscribe/unsubscribe to dataset rooms
- `use-import-socket.ts` - Import progress events
- `use-column-config-socket.ts` - Column config change events
- `use-normalization-socket.ts` - Normalization progress events
- `use-validation-socket.ts` - Email validation progress events
- `use-webhook-socket.ts` - Webhook data received events
- `features/batch-enrichment/hooks/useEnrichmentSocket.ts` - Enrichment progress
- `features/notifications/hooks/useEnrichmentNotifications.ts` - Notification events

**Socket-to-state flow:**
1. Component mounts, registers listener via `socketService.on()` or `socketService.onAny()`
2. Server emits event with payload
3. Hook callback updates local state or invalidates React Query cache
4. Component re-renders with updated data

## Patterns
- Context for identity (who am I, which workspace), React Query for data (what data do I have)
- Socket events trigger React Query invalidation rather than maintaining parallel state
- Workspace is a hard boundary: switching clears all cached data
- Auth token is stored in both localStorage and the ApiClient instance
- HMR-safe context pattern: store context object in `globalThis` to survive module replacement

## Undocumented
- `use-feature-check.ts` is a stub that always returns `{ enabled: true, loading: false }` with a TODO comment about wiring to a backend endpoint
- The socket listener registry survives disconnect/reconnect cycles by design (stored in `listenerRegistry` array)
- `use-user-initialization` has a 3-second delay before checking if the user needs default folder creation
- Two toast systems coexist: `use-toast.ts` (Radix-based) is used by most hooks; `sonner` is used by user initialization

## Concerns
- `use-feature-check` is always-true, meaning there is no actual feature gating in the frontend
- Workspace switch calls `queryClient.clear()` which is destructive; a more targeted invalidation could preserve non-workspace data
- The socket `listenerRegistry` grows over time and is only cleaned when individual listeners call their cleanup function; there is no global cleanup
- No optimistic updates are used in mutations despite React Query supporting them
- The `isAuthenticated` state in AuthContext has no server-side validation interval; a revoked session stays "authenticated" until the next API call returns 401
