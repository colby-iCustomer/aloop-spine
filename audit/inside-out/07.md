### Connection Flows

**OAuth Flow** (LinkedIn, Meta, Reddit, Google, HubSpot):
1. Frontend builds OAuth URL: `{GATEWAY_BASE_URL}/api/integrations/{provider}/auth?userId=X&workspaceId=Y`
2. User redirected to provider's consent screen
3. Callback returns to gateway, which stores tokens
4. Frontend polls for new connected account
5. For LinkedIn/Meta: additional Airbyte connection setup call (`POST /api/integrations/{provider}/connection/setup`)

**API Key Flow** (Smartlead, Apollo, Klaviyo, Mailshake):
1. User enters API key in credentials dialog
2. Frontend validates: `POST /api/integrations/{provider}/validate-key`
3. On success: `POST /api/integrations/{provider}/connect`
4. Account appears in connected accounts list

**Connection Provider Flow** (Salesforce):
1. Uses Polytomic adapter pattern
2. Frontend calls `/connections/*` endpoints
3. Separate `SalesforceImportWizard` component handles the import flow
4. Dedicated service: `salesforceDataService` and `connectionsService`

## Patterns
- Registry pattern: provider configs are looked up by integration name string
- All integration API calls go through gatewayClient, not apiClient
- Export flows are lazy-loaded components inside the grid's action drawer
- Connected account normalization: raw snake_case API responses -> camelCase frontend types
- Category system: "Marketing & Ad Platforms", "CRM & Customer Data Platforms", "Databases & Data Sources"

## Undocumented
- Polytomic is the adapter layer for Salesforce (and potentially Instantly, Marketo) but the name "Polytomic" only appears in type names and utility files
- The `polytomicOAuth.ts` utility suggests Polytomic handles its own OAuth flow separate from the main OAuth flow
- LinkedIn Ads has a `preLogout` flag that triggers a pre-logout call before OAuth redirect
- Apollo integration has a "system" provider concept where a system-managed API key is auto-resolved
- `INTEGRATION_NAME_OVERRIDES` maps backend inconsistencies: "reddit" -> "Reddit", "smartleads" -> "Smartlead"

## Concerns
- Instantly and Marketo are listed in `connectionProviders.ts` but have no provider registry entries, catalog entries, or UI components
- The provider registry uses exact string matching on integration names (e.g., "LinkedIn Ads"), which is fragile
- Export flows are 9 separate components with likely significant code duplication (each handles account selection, field mapping, confirmation)
- The `audiencesService` is deprecated but its replacement (`live-audiences/services.ts`) only covers LinkedIn and Meta; other platform audience management may be incomplete
