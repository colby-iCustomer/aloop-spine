# 10 - Dev Workflow

## What Docs Say

### Git Conventions

**Branching:**
- `main` = stable + Lovable commits (Lovable always commits to main)
- `feature/{name}` = code-edit work (auth, routing, API stubs)
- Squash merge to main after code review
- Delete feature branch after merge
- Before each major feature: create branch from latest main

**Commit Format:** `[category] description`
- Categories: `feat`, `fix`, `refactor`, `design`, `docs`
- Descriptions: concise, describe what changed, not why
- Examples: `[feat] Add Supabase Auth with Google OAuth`, `[fix] StatusBadge - use success token instead of green-600`

**Pre-commit checks:**
- Run `bun run build` to verify no TypeScript errors
- Check that no hardcoded hex colors or raw Tailwind color literals were introduced

### Build System

- **Bun** for package management and build
- `bun run build` - production build with TypeScript checking
- Vite 5 as the build tool
- Production output to `dist/`

### Lovable Compatibility

The repo is iterated locally and synced back to Lovable:
- Avoid non-standard build plugins or Vite config changes
- Keep component structure flat and readable (Lovable's editor parses JSX)
- shadcn/ui is the preferred primitive layer, no other UI libraries
- `tmp/` folder for scratch files during migration (not committed)

### Planning System (Superpowers)

**Active specs** in `docs/superpowers/specs/` - brainstorming outputs/design specs
**Active plans** in `docs/superpowers/plans/` - implementation step plans
**Archive** in `docs/archive/plans/` - pre-superpowers era plans

Completed plans automatically generate a completion entry in `docs/current-repo/decisions/` with:
- Decisions made during execution (deviations from plan)
- Design token / code quality fixes
- Post-plan additions
- Old app critique
- Final file inventory
- Lessons learned

### Working Docs System

Living reference documents in `docs/current-repo/working-docs/`:
- **API Endpoint Map** - every endpoint with validation status
- **Styling Decisions** - visual/typographic decisions with alternatives tested
- **Troubleshooting** - known issues with index and per-issue files

Rules: check before touching a domain, update when making decisions, update when reverting.

### Port Feature Pipeline

Multi-agent pipeline for porting features from old app:
- `agents/gatherer-prompt.md` - context gathering
- `agents/frontend-prompt.md` - implementation
- `agents/reviewer-prompt.md` - code review
- `agents/wiring-prompt.md` - API wiring
- `agents/spot-check-prompt.md` - verification

Each feature port produces a log in `docs/port-feature-pipeline/logs/{NN}-{feature}/` with context, spec, review, and wiring decisions.

### Decision Journal

`docs/current-repo/decisions/` contains 25+ post-completion entries covering architectural decisions, API wiring notes, and plan deviations. Each entry linked to its source plan.

### PR Retrospectives

`docs/prs/` contains session retrospectives with context for major PRs:
- chat-preamble, chat2-list-view-summary, chat-3-folder-modals-summary
- Per-feature retros (templates-page, batch-enrichment)
- Backend coordination docs (dinesh-tickets)

### Testing

- No testing framework or test files documented
- `bun run build` is the only documented verification step
- No CI/CD pipeline documented
- No linting configuration documented (beyond TypeScript compilation)

### Deployment

- Lovable handles production builds/deployment for the main app
- No Vercel deployment documented for this repo specifically
- No staging environment documented

## Regression Flags
- Old repo had **highlight.run** for error tracking. No error tracking documented for new repo.
- Old repo had **PostHog** for analytics. No analytics tracking documented for new repo.
- Old repo had **console.log with emoji in production** in multiple files. CLAUDE.md does not explicitly prohibit console.log statements (only prohibits hardcoded colors and em dashes).
- Old repo had **no lazy loading** which was a known performance issue. New repo has all routes lazy-loaded, which is an improvement.
- Old repo had **no React Query** (manual useEffect/useState fetching with no caching). New repo uses React Query throughout, which is an improvement.
- The old repo code review identified that `useOrganization.inviteMember` was a stub that showed success toast but did nothing. Verify new repo invitation hooks actually work.
