# 06 - State Management

## What Docs Say

### Primary State Patterns

**1. React Context - Identity and Workspace**

`AuthContext` (`src/contexts/AuthContext.tsx`) is the primary state holder for:
- User identity (`UserProfile` with email, name, avatar, role)
- Active workspace (`workspaceId`, `workspaceName`, `workspacePermissions`)
- Auth operations (`loginWithGoogle`, `logout`, `switchWorkspace`)

`WorkspaceContext` mentioned in some docs as a separate context for workspace resolution, but the workspace-management flow doc shows workspace state lives directly in AuthContext's `user` object.

**2. React Query - Server State**

All data fetching uses React Query with:
- Workspace-scoped query keys: `["folders", wsId]`, `["folder", wsId, id]`, `["workspace", wsId]`, `["team", wsId]`, `["pixel-status", wsId]`
- User-scoped query keys (no workspace): `["subscription"]`, `["profile"]`, `["credit-breakdown"]`
- `enabled: !!wsId` pattern prevents queries from firing before auth resolves
- `staleTime` varies: folders 2 min, workspaces 5 min, team 60s

**3. Module-Level Singletons - Performance-Critical State**

For enrichment rendering, state is kept OUTSIDE React to avoid re-renders:
- `cellStatusTracker` (`src/lib/cell-status-tracker.ts`) - singleton Map tracking per-cell enrichment status as `"rowId:columnId"` keys
- Enrichment timer - module-level `startEnrichmentTimer()` / `stopEnrichmentTimer()` / `getEnrichmentTimerState()`
- `cellEditQueue` (`src/lib/cell-edit-queue.ts`) - debounced PATCH persistence for user edits

**4. localStorage - Persistent State**

| Key | Value | Purpose |
|-----|-------|---------|
| `jwt_token` | JWT string | API authentication |
| `user_id` | User ID | Session identity |
| `currentWorkspaceId` | Workspace ID | Persist active workspace across reloads |
| `has_logged_in_before` | `"true"` | First-login detection |
| `columnMapping_{listId}_{templateId}` | JSON | Template field mapping cache |

**5. sessionStorage - Transient State**

| Key | Value | Purpose |
|-----|-------|---------|
| `pendingTemplateId` | Template UUID | Cross-page template activation flow |
| `pendingTemplateName` | Template name | Cross-page template activation flow |

**6. URL State**

- `?mappingId=xyz` - template context for list view
- `?createnewlist=true` - auto-open create list modal
- `?partner=CODE` - referral code
- React Router `state` object: `templateMapped`, `templateId`, `mappingId`

### Workspace Switch Cache Strategy

On workspace switch, `queryClient.clear()` wipes the ENTIRE React Query cache. This is a deliberate tradeoff:
- All workspace-scoped data (folders, team) is evicted
- All user-scoped data (subscription, credits) is ALSO evicted but refetches transparently
- First page render after switch may show brief loading states for non-workspace-scoped data

### Enrichment State Architecture

The enrichment system uses a strict separation:
- **User edits**: `onCellValueChanged` -> `cellEditQueue` -> PATCH API (React pathway)
- **Enrichment updates**: socket events -> `cellStatusTracker` -> `applyTransaction()` (non-React pathway)
- These pathways are structurally isolated. `applyTransaction()` does NOT fire `onCellValueChanged`, making it impossible for enrichment to reach the edit queue.

The `cellStatusTracker` uses a publish/subscribe pattern with rAF-batched notifications. Subscribers (like `useEnrichmentRefresh`) receive batched updates and call `gridApi.refreshCells()` to trigger cell renderer re-reads.

### Hook Patterns

- `useWorkspaceId()` - thin extractor returning `user?.workspaceId ?? ""`
- `useHasPermission(permission)` - two-tier: API permissions array (primary) or role fallback
- `useModalState(keys)` - replaces multiple `useState(false)` for modal boolean clusters
- `usePlanCapabilities()` - mentioned in RBAC spec for plan-gated features (uses `useSubscription()`)
- `useFeatureCheck()` - pass-through stub for feature flags (`enabled: true` always)

### Toast/Notification State

Two systems running side-by-side:
- **Sonner** (preferred) - `import { toast } from "sonner"` - clean pill notifications
- **shadcn Radix toast** (legacy) - `import { toast } from "@/hooks/use-toast"` - wider box style
- Migration plan: move all to Sonner, remove Radix toast infrastructure

## Regression Flags
- Old repo used **Jotai atoms** for global state (JWT, user, workspace, projects). New repo uses React Context + React Query. Migration documented as complete.
- Old repo had `CreditsProvider` wrapping entire app with subscription fetch, credit consumption, overage modals, and plan restrictions. New repo replaces with `useSubscription()` + `usePlanCapabilities()` hooks. Verify provider is removed.
- Old repo had `TemplateHeaderProvider` for grid template state. New repo should handle this in `features/templates/` context. Verify.
- Old repo had **no React Query / TanStack Query**. All fetching was manual `useEffect` + `useState`. The old repo retro specifically called this out as needing improvement.
- Old repo had **per-row enrichment status on row data** (8+ magic-prefixed properties like `_isEnriching`, `_cellStatus_{col}`). New repo uses separate `cellStatusTracker` singleton. Verify no row decoration remains.
- Old repo had **no request deduplication** across hooks (only `useCredits` had it). React Query handles this automatically now.
- Old repo had `dummyProgress` state for enrichment notification progress blending (0-15% fake, then real). Verify this is implemented in new notification module.
