# 09 - Permissions and RBAC

## What Docs Say

### Role Hierarchy

```
Owner > Admin > Editor > Viewer
```

Note: the codebase `UserRole` type defines `"owner" | "admin" | "member" | "viewer"`. The `member` role must be renamed to `editor` per the RBAC spec. Both entries exist in the `ROLE_PERMISSIONS` map for backward compatibility.

### Role Descriptions

| Role | Description |
|------|-------------|
| Owner | Workspace billing owner. One per workspace. Creates additional workspaces (Pro). Cannot be removed or demoted. |
| Admin | Workspace team manager. Invites members, changes roles, manages workspace settings. |
| Editor | Audience creator. Builds, edits, and exports audiences. Cannot manage team. |
| Viewer | Read-only access. Views audiences and settings. Cannot create, edit, or export. |

### Permissions Matrix

| Permission | Owner | Admin | Editor | Viewer |
|---|:---:|:---:|:---:|:---:|
| View audiences, team, settings | Y | Y | Y | Y |
| Create and edit audiences | Y | Y | Y | - |
| Delete folders and lists | Y | Y | Y | - |
| Export data | Y | Y | Y | - |
| Invite and remove members | Y | Y | - | - |
| Change member roles | Y | Y | - | - |
| Manage workspace settings | Y | Y | - | - |
| Allocate credits and overage | Y | - | - | - |
| Create workspaces (Pro) | Y | - | - | - |
| Manage billing and plan | Y | - | - | - |
| Delete workspace | Y | - | - | - |
| Leave workspace (self-remove) | - | Y | Y | Y |

### Permission Hook: `useHasPermission(permission)`

Two-tier resolution:
1. **API permissions (primary)**: if `user.workspacePermissions.length > 0`, check array inclusion
2. **Role fallback**: if permissions empty, derive from `user.role` using static `ROLE_PERMISSIONS` map

Permission strings used in frontend:

| Permission String | Maps To |
|---|---|
| `workspace.manage` | Invite, remove, change roles, edit settings |
| `folder.view` | View folders |
| `folder.create` | Create folders |
| `folder.edit` | Edit folders |
| `folder.delete` | Delete folders |
| `list.create` | Create lists |
| `list.edit` | Edit lists |
| `list.delete` | Delete lists |
| `data.export` | Export data |
| `credits.allocate` | Allocate credits and overage (Owner only) |

### User Types

**Full User** - business email domain, own workspace on signup, all roles available.

**Guest Viewer** - personal email domain (gmail, yahoo, etc.), dormant personal workspace (Viewer in own workspace), can only be Viewer in shared workspaces, Pro plan invitation only. Dormant workspace is hidden from workspace selector.

### Plan Gating

Three plan tiers with feature gating:

| Feature | Free | Starter | Professional |
|---------|------|---------|-------------|
| Team members | 1 | 3 | 10 |
| Templates | 3 specific UUIDs | All | All |
| Integrations | 5 specific names | All | All |
| Pixels and Tags | No | Yes | Yes |
| Multiple workspaces | No | No | Yes |

**Critical issue**: Plan capabilities are hardcoded in frontend. No API endpoint exposes "what can this plan access?" This means:
- Frontend is the ONLY enforcer for free-plan restrictions
- MCP callers cannot check feature access without hardcoding same values
- If template UUIDs or integration names change, frontend breaks silently

### Workspace Authorization Security Gap

**CRITICAL**: No `X-Workspace-Id` header middleware exists. All workspace-scoped endpoints must validate workspace membership before processing. Without this, users can access resources across workspaces by ID. The frontend navigates to `/` on workspace switch as a stopgap but this is not a security boundary.

### Feature Flags

`useFeatureCheck()` hook exists but is a pass-through stub (`enabled: true` always). Waiting for `GET /features/{key}/check` backend endpoint.

### Workspace Creation Gating Discrepancy

The sidebar allows workspace creation regardless of plan (no check). The `WorkspaceSelector` enforces Professional plan gating. The `WorkspaceSelector` variant is the intended gated behavior.

### RBAC Audit (from ALOOP Audit doc)

The audit identified these implementation gaps:

**Sidebar:**
- `folder.view` gate: missing (always shown)
- `folder.edit`/`folder.delete` gate on 3-dot menu: missing
- `workspace.manage` gate on Manage Users nav: missing
- `canAccessPixelsAndTags()` on Pixels and Tags nav: missing

**Folder Page:**
- Various list.create, list.delete gates: status unclear from audit doc alone

**ListView:**
- `list.edit` controls 7 UI elements (edit button, delete rows, SheetOps, Enrich, Actions, grid editing): partially implemented

- No documentation of JWT token contents or claims structure

- Feature flag system is a placeholder with no real backend
- No documentation of audit logging or access tracking


## Regression Flags
- Old repo had **PrivateRoute** component for permission-gated routing (redirected to `/` with toast if denied). Not implemented in new repo.
- Old repo had **Layer 2 app roles** via Supabase (`useRoles` hook, `user_roles` table). This was a legacy parallel system. Verify it is NOT carried forward.
- Old repo had `hasPermission()` on `useWorkspaces()` hook. New repo has `useHasPermission()` as a separate hook.
- Old repo had **resource sharing** endpoints (`GET/POST/DELETE /workspaces/:id/shares/*`). Not mentioned in new repo RBAC.
- Old repo had `leaveWorkspace()` - `POST /workspaces/:id/leave`. Listed as missing endpoint in new repo.
- Old repo had workspace invitation listing: `GET /workspaces/:id/invitations`. New repo uses `POST /invitations` (body-scoped, not path-scoped).
- The RBAC audit doc identifies multiple missing permission gates in the new repo sidebar, folder page, and list view. These need code verification.
