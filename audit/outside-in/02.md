# 02 - Architecture

## What Docs Say

### Application Structure

The app is a React SPA built with Vite, organized into a feature-based architecture. The `src/` directory has these top-level divisions:

| Directory | Purpose |
|-----------|---------|
| `api/` | HTTP client, service methods, React Query hooks, TypeScript types (`api/types/`), mock data (`api/mocks/`) |
| `components/auth/` | Auth pages: logo, marketing panel, Google OAuth button, `ProtectedRoute` |
| `components/common/` | Shared UI: `StatusBadge`, `PageHeader`, `EmptyState` |
| `components/layout/` | App shell: `AppLayout`, `AppSidebar`, `TopBar`, `CreditsPopover` |
| `components/ui/` | shadcn/ui primitives (50+ components) |
| `components/folder/` | Folder-page modals (CSV upload, Google Sheets, HubSpot, Webhooks) |
| `components/list/` | List-page modals (ICP prospecting, Klaviyo, web search agent) |
| `components/settings/` | Settings tabs (General, API Keys, Billing, Integrations, Security, etc.) |
| `components/credits/` | Credits widget and overage confirmation |
| `contexts/` | `AuthContext` (identity), `WorkspaceContext` (workspace state/resolution) |
| `features/` | Feature modules (self-contained: types, hooks, components, stubs) |
| `hooks/` | Shared hooks (`use-mobile`, `use-modal-state`, `use-toast`) |
| `lib/` | Utility functions (`utils.ts`) |
| `pages/` | All page components (lazy-loaded from `App.tsx`) |

### Feature Module Pattern

Domain-specific code lives in `src/features/{feature}/`, each self-contained with:
- `types.ts` - domain types
- `hooks/` - data fetching and business logic hooks
- `components/` - UI components
- `services/` - API service methods (feature-local)
- `utils/` - feature-specific utilities
- `stubs.ts` - mock data for development
- `index.ts` - barrel export

Documented feature modules:
- `grid/` - AG Grid configuration, hooks (useGridData, useGridColumns, useGridEditing, useGridPaste, useGridRows, useGridSelection), cell renderers, toolbar, enrichment UI
- `enrichment/` - Enrichment hooks (useEnrich, useCancelEnrichment, useFillColumn), socket handling
- `templates/` - Template definitions, mapping services, field mapping, sample data service
- `live-audiences/` - Live audience queries, services, platform tabs
- `integrations/` - OAuth polling, API key flow utilities, integration UI
- `notifications/` - Novu integration, notification bell, popover

### Data Flow Architecture

```
API Layer (services.ts, feature services)
  -> React Query hooks (hooks.ts, feature hooks)
    -> Page components (pages/)
      -> Feature components (features/*/components/)
        -> UI primitives (components/ui/)
```

### Provider Nesting (current app)
```
<GoogleOAuthProvider>
  <QueryClientProvider>
    <BrowserRouter>
      <NovuProviderWrapper>
        <AuthProvider>
          <WorkspaceProvider>  (or workspace state in AuthContext)
            <Routes />
          </WorkspaceProvider>
        </AuthProvider>
      </NovuProviderWrapper>
    </BrowserRouter>
  </QueryClientProvider>
</GoogleOAuthProvider>
```

### Service Topology

Two backend services:
1. **list-service** (`https://list-service.icustomer.ai/api/v1`) - Projects, lists, data, columns, enrichment, payments, auth
2. **gateway** (`https://prod-aloop-api-gateway.icustomer.ai`) - Integration OAuth, exports, analytics, prospecting

All API responses follow `{ success: true, data: T, pagination?: {...} }` or `{ success: false, error: string }`.

### Client Architecture
Single `ApiClient` class (fetch-based) with token management:
- `apiClient` - talks to list-service
- `gatewayClient` - talks to gateway
- Both share token state via `apiClient.setToken()` called from AuthContext

### Enrichment Data Pipeline
```
User clicks Enrich -> REST POST /lists/:listId/enrich
  -> Bull Queue (Redis) -> Worker picks job
  -> Streams to Python Agent (SSE)
  -> Row-by-row results via SSE events
  -> Socket.IO emits to subscribed FE clients
  -> AG Grid applyTransaction() updates individual rows
```

### Real-time Architecture
Two separate socket event channels:
- **Grid events** (underscore format): `row_enriched`, `enrichment_progress`, `row_enrichment_queued/processing/completed` - update cell data
- **Notification events** (colon format): `enrichment:started/progress/completed/failed/cancelled` - update progress UI

### Mock Data Strategy
- `VITE_USE_MOCKS=true` toggle for development
- Mock data in `api/mocks/` and feature `stubs.ts` files
- Stub to API swap: create `services.ts` with same signatures, change imports in hooks, delete stubs

## Regression Flags
- Old repo had **5 different HTTP client abstractions** (fetchWithAuth, apiClient, paymentClient, fetchIntegrationsAPI, supabase). New repo consolidated to 2 (apiClient, gatewayClient). Verify no endpoints still use legacy patterns.
- Old repo had **Supabase direct queries** from hooks (useOrganization, useDatasets). Should be fully removed.
- Old repo had **Jotai atom cascade** for workspace state propagation. New repo uses explicit invalidation via React Query. The workspace data scoping doc identified incomplete wiring (service fallback to default workspace). Verify this was fixed.
- Old repo had **deep provider nesting** (10 levels). New repo should be flatter.
- Old repo had **no lazy loading** on routes. New repo has all 14 routes lazy-loaded.
- Old repo **ListView was 2,954 lines** with 35+ useState calls. New repo decomposed into feature hooks. Verify decomposition is complete.
